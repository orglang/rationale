# Что такое трансакция

> *"Видишь суслика? — Нет. — И я не вижу. А он есть!"*
> <p align="right">— ДМБ (2000)</p>

Попробуем увидеть ~~суслика~~трансакцию на айтишных примерах, но придется приложить усилия.

> [!WARNING]
> Несмотря на то, что *транзакция* является устоявшимся понятием в IT-индустрии, мы будем использовать очень схожий термин *трансакция*, но в абсолютно другом значении.

## Объяснение на пальцах

Начнем с примера, в котором точно нет трансакций.

```go
func DoSomeWork() WorkOutcome {
    outcome := DoFirstTask()
    return DoSecondTask(outcome)
}
```
<p align="center"><sub>Листинг 1. Чистая работа</sub></p>

Считаем, что `DoFirstTask` и `DoSecondTask` являются чистыми функциями без побочных эффектов.

> [!IMPORTANT]
> Нет побочных эффектов => нет трансакций

Рассмотрим следующий пример.

```go
func DoSomeWork() WorkOutcome {
    provider := NewWorkProvider()
    outcome := provider.DoFirstTask()
    return provider.DoSecondTask(outcome)
}
```
<p align="center"><sub>Листинг 2. Работа через провайдера</sub></p>

В примере из листинга 1 функция `DoSomeWork` была чистой. Теперь же в рамках неё создается провайдер, которому делегируется выполнение работы. Данная операция является побочным эффектом, т.к. при её выполнении аллоцируются вычислительные ресурсы. Можно ли такого рода операцию назвать трансакцией? Ответ зависит от реализации функции `NewWorkProvider`.

Для яркости примера будем считать, что при создании провайдера аллоцируются сотни гигабайт оперативной памяти.

Если свободная оперативная память уже в наличии или она добавляется автоматически по запросу, то трансакции нет. Это значит, что всё случилось до начала работы программы.

> [!IMPORTANT]
> Аллокация ранее согласованных ресурсов не является трансакцией.

Если в рамках вызова `NewWorkProvider` (чисто гипотетически) программа приостанавливается, ищется владелец требуемого количества оперативной памяти, выполняется обоснование, фиксируются договоренности, память физически добавляется на машину, затем программа возобновляется, то это и есть трансакция.

Трансакция - это процедура получения доступа к ресурсу, процедура вовлечения ресурса в работу. Обычно такая процедура осуществляется до начала работы программы.

Основная проблема с трансакциями в том, что они несут издержки. Трансакция - это как правило ручные действия, а ручные действия чаще всего ведут к очень большим издержкам (еще рискам и неопределенностям, но об этом как-нибудь в другой раз).

Проблема понятна, но как её решать?

Нужен инструмент, который позволит увидеть трансакции, как микроскоп позволяет увидеть микромир. Это как минимум даст возможность посчитать трансакционные издержки, поставить их на учет. Также нужен инструмент, который позволит замещать тяжеловесные трансакции с участием людей на легковесные трансакции с участием ботов.

> [!IMPORTANT]
> Orglang является инструментом, который позволяет увидеть трансакции, посчитать их издержки, спроектировать и заместить их более легковесными аналогами.

В идеале все трансакции выполняются в автоматическом режиме без ручного вмешательства (снятие трансакций). Трансакционные издержки снижаются на порядки. Вместе с этим уходят сопутствующие им риски и неопределенности.

## Объяснение формальное

За основу возьмем формальную модель теории сессионных типов [^1]. В рамках теории существует операция порождения нового процесса, которая является аналогом оператора `new` в языках программирования. В ванильной теории операция порождения является тривиальной, т.к. делается допущение о том, что все ресурсы доступны. Нам же придется копнуть чуть глубже и определить то, что происходит под капотом.

### Плейсхолдеры и идентификаторы

> [!WARNING]
> Вместо устоявшегося в промышленных языках программирования понятия *переменной* будем использовать термин *плейсхолдер*, т.к. он лучше передает семантику одноразовости подстановки.

Плейсхолдеры, которые мы будем записывать как *z*, *y*, *x* (латинские символы конца алфавита), являются частью синтакиса программ и обретают значение во время выполнения при подстановке. Ключевое отличие от функциональных ЯП, где на место плейсхолдера подставляется целое выражение, заключается в том, что мы вместо этого будем подставлять *идентификатор*. Идентификаторы, которые мы будем записывать как *a*, *b*, *c* (латинские символы начала алфавита), могут представлять разное в зависимости от интерпретации. Например, они могут представлять адреса в памяти, каналы коммуникации между процессами или, как в нашем случае, трансакции между пулами.

### Термы, значения и продолжения

Сопокупность термов пулов представляет состояние хозяйствующего субъекта, осуществляющего некоторую деятельность. Значения и продолжения пула представляют два разных вида информации. Значения - это небольшие (фиксированного размера) кусочки данных, тогда как продолжения - это данные, которые могут включать терм пула целиком.

Отличительная особенность заключается в том, что термы можно исполнять (execute), значения - разбирать (observe), а продолжения - применять (apply).

Формально пулы, значения и продолжения имеют грамматику, которая представлена в таблицах 1, 2 и 3.

| Значение *V*  | Семантика |
| :-----------  | :-------- |
| close          | Терминальный токен без какой-либо полезной информации. |
| shift(*c*<sub>*k*</sub>) | Трансакция *c*<sub>*k*</sub> в режиме *k*, помеченная как shift. |
| choice(*c*) { *l* }   | Трансакция *c*, помеченная как choice, с классом *l*. |
| coop(*c*) { *w*·*q* } | Трансакция *c*, помеченная как coop, с долей *w* в производительности по именованному процессу *q*. |
| isol(*c*) { *w*·*q* } | Трансакция *c*, помеченная как isol, с долей *w* в производительности по именованному процессу *q*. |
| spawn(*c*)&nbsp;{&nbsp;*a*&nbsp;←&nbsp;*q*&nbsp;*b*<sub>1</sub>&nbsp;*b*<sub>2</sub>&nbsp;...&nbsp;*b*<sub>*n*</sub>&nbsp;} | Трансакция *c*, помеченная как spawn, с вызовом именованного процесса *q* и аргументами *a*, *b*<sub>1</sub>, *b*<sub>2</sub>,...,*b*<sub>*n*</sub>. |
<p align="center"><sub>Таблица 1. Грамматика для значений</sub></p>

| Продолжение *K* | Семантика |
| :-------------- | :-------- |
| (close ⇒ *P*) | При получении терминального значения close продолжить как *P*. |
| (shift(*x*<sub>*k*</sub>) ⇒ *P*) | При получении значения shift(*c*<sub>*k*</sub>) продолжить как *P*[*c*<sub>*k*&nbsp;</sub>/*x*<sub>*k*&nbsp;</sub>]. Происходит связывание плейсхолдера *x*<sub>*k*</sub> в рамках терма *P*. |
| (choice(*x*<sub>*𝚓*</sub>) { *𝚓* } ⇒ *P*<sub>*𝚓*</sub>)<sub>&nbsp;*𝚓* ∈ *𝙹*</sub> | При получении значения choice(*c*){*l*} для некоторого класса *l* ∈ *𝙹* продолжить как *P*<sub>*l*&nbsp;</sub>[*c*/*x*<sub>*𝚓*</sub>]. Происходит связывание плейсхолдера *x*<sub>*𝚓*</sub> в рамках терма *P*<sub>*𝚓*&nbsp;</sub> для каждого паттерна *𝚓* ∈ *𝙹*. |
| (coop(*x*) { *w*·*q* } ⇒ *P*) | При получении значения coop(*c*){*w*·*q*} продолжить как *P*[*c*/*x*]. Происходит связывание плейсхолдера *x* в рамках терма *P* и наращивание доли в производительности *q*. |
| (isol(*x*) { *w*·*q* } ⇒ *P*) | При получении значения isol(*c*){*w*·*q*} продолжить как *P*[*c*/*x*]. Происходит связывание плейсхолдера *x* в рамках терма *P* и сокращение доли в производительности *q*. |
| (spawn(*x*)&nbsp;{&nbsp;*z*&nbsp;←&nbsp;*q*&nbsp;*y*<sub>1</sub>&nbsp;*y*<sub>2</sub>&nbsp;...&nbsp;*y*<sub>*n*</sub>&nbsp;}&nbsp;⇒&nbsp;*P*) | При получении значения spawn(*c*){*a* ← *q* *b*<sub>1</sub> *b*<sub>2</sub>&nbsp;...&nbsp;*b*<sub>*n*</sub>} для некоторого *q* ∈ *Q* продолжить как *P*[*c*/*x*]. Происходит связывание плейсхолдера *x* в рамках терма *P* и порождение процесса *q*. |
<p align="center"><sub>Таблица 2. Грамматика для продолжений</sub></p>

| Терм пула *P* | Семантика |
| :-------------- | :-------- |
| *c* ← *a* | Соединить каналы *a* и *c*. |
| *c*.*V* | Коммуницировать значение *V* в канал *c*. |
| case *c* *K* | Коммуницировать продолжение *K* в канал *c*. |
<p align="center"><sub>Таблица 3. Грамматика для термов</sub></p>

Где *x*, *x*<sub>*𝚓*</sub>, *z*, *y*<sub>1</sub>, *y*<sub>2</sub>,... представляют *плейсхолдеры* каналов, тогда как *a*, *b*, *c*, *b*<sub>1</sub>, *b*<sub>2</sub>,... представляют *идентификаторы* каналов. *k* представляет *режимы* каналов.

Ключевой операцией языка является подача значения в продолжение, в результате которой образуется терм. Продолжение содержит терм пула (либо выбор между несколькими термами) со связанными плейсхолдерами в рамках него, и значение может как выбрать подходящий терм пула, так и предоставить идентификатор для подстановки.

Будем записывать данную операцию как *V* ▷ *K* и определим её следующим образом:  
> close⠀▷⠀(close ⇒ *P*)⠀≜⠀P  
> shift(*c*<sub>*k*</sub>)⠀▷⠀(shift(*x*<sub>*k*</sub>) ⇒ *P*)⠀≜⠀*P*[*c*<sub>*k*&nbsp;</sub>/*x*<sub>*k*&nbsp;</sub>]  
> choice(*c*){*l*}⠀▷⠀(choice(*x*<sub>*𝚓*</sub>){*𝚓*} ⇒ *P*<sub>*𝚓*</sub>)<sub>&nbsp;*𝚓* ∈ *𝙹*</sub>⠀≜⠀*P*<sub>*l*&nbsp;</sub>[*c*/*x*<sub>*j*</sub>]  
> coop(*c*){*w*·*q*}⠀▷⠀(coop(*x*){*w*·*q*} ⇒ *P*)⠀≜⠀*P*[*c*/*x*]  
> isol(*c*){*w*·*q*}⠀▷⠀(isol(*x*){*w*·*q*} ⇒ *P*)⠀≜⠀*P*[*c*/*x*]  
> spawn(*c*){*a* ← *q* *b*<sub>1</sub> *b*<sub>2</sub> ... *b*<sub>*n*</sub>}⠀▷⠀(spawn(*x*){*z* ← *q* *y*<sub>1</sub> *y*<sub>2</sub> ... *y*<sub>*n*</sub>} ⇒ *P*)⠀≜⠀*P*[*c*/*x*]   

Для остальных комбинаций *V* и *K* операция *V* ▷ *K* остается неопределенной.

### Интерпретация

Мы интерпретируем идентификаторы как приватные трансакции между пулами, в рамках которых осуществляется обмен сообщениями. Пул 1 может опубликовать значение и/или подписать продолжение в рамках трансакции, так что данные сообщения будут получены пулом 2 на другом конце трансакции. Значение *V* может быть опубликовано в трансакцию *a* при помощи конструкции *a*.*V*. Данное сообщение затем будет получено подпиской case *a* *K*, которая подаст полученное значение *V* в продолжение *K* для определения дальнейшего развития событий.

> *y*<sub>1</sub> : *B*<sub>1</sub>,...,*y*<sub>*n*</sub> : *B*<sub>*n*</sub> ⊢ *P* :: (*z* : *A*)

Пул *P* может *использовать* трансакции *y*<sub>1</sub>,...,*y*<sub>*n*</sub> для коммуникации с другими пулами по протоколам *B*<sub>1</sub>,...,*B*<sub>*n*</sub> соответственно и *предоставляет* трансакцию *z* по протоколу *A*, в рамках которой с ним могут коммуницировать другие.

| Тип | Семантика |
| :-------------- | :-------- |
| 𝟏  | Отправить токен close без продолжения. |
| ￪ *A* | Отправить токен accept. Продолжить как *A*. |
| ￬ *A* | Отправить токен detach. Продолжить как *A*. |
| ⨁{ *l* : *A*<sub>*l*</sub> }<sub> *l* ∈ *L*</sub> | Отправить токен choice с классом *l* ∈ *L*. Продолжить как *A*<sub>*l*</sub>. |
| &{ *l* : *A*<sub>*l*</sub> }<sub> *l* ∈ *L*</sub> | Принять токен choice с классом *l* ∈ *L*. Сопоставить класс с паттерном и продолжить как *A*<sub>*l*</sub>. |
| ⮥ <sub>*w*·*q* ∉ *Q*</sub> [*Q*'] *A* | Отправить токен coop с именем процесса *q* и долей в производительности *w* такими, что *w*·*q* ∉ *Q*. Вычислить объединение *Q*' = *Q* ⋃ {*w*·*q*}. Продолжить как [*Q*'] *A*. |
| ⮧ <sub>*w*·*q* ∈ *Q*</sub> [*Q*'] *A* | Принять токен isol с именем процесса *q* и долей в производительности *w* такими, что *w*·*q* ∈ *Q*. Вычислить пересечение *Q*' = *Q* ⋂ {*w*·*q*}. Продолжить как [*Q*'] *A*. |
| *q*(*a*,&nbsp;*b*<sub>1</sub>,&nbsp;*b*<sub>2</sub>,...,*b*<sub>*n*</sub>)&nbsp;⊸<sub>*q*&nbsp;∈&nbsp;*Q*</sub>&nbsp;*A*<sub> | Принять токен spawn с именем процесса *q* и каналами процессов *a*, *b*<sub>1</sub>, *b*<sub>2</sub>,...,*b*<sub>*n*</sub>. Продолжить как *A*. |
<p align="center"><sub>Таблица 4. Типы трансакций</sub></p>

Где *Q* представляет мультимножество имен процессов, кратность *w* в котором представляет долю в производительности { *w*<sub>1</sub>·*q*<sub>1</sub>, *w*<sub>2</sub>·*q*<sub>2</sub>,...,*w*<sub>*n*</sub>·*q*<sub>*n*</sub> }.

### Примеры

```
t0 = &{
    task : q(a, b1, b2,...) ?[1·q ∈ Q] ⊸ ![Q − {1·q}] t0,
    more : t1,
    less : t2,
    stop : ?[Q = {}] ￬ 𝟏
}

t1 = ￪ (?[w·q ∉ Q] ⮥ ![Q ⋃ {w·q}] t0)

t2 = ￪ (?[w·q ∈ Q] ⮧ ![Q ⋂ {w·q}] t0)
```

Где исходно *Q* = {}.

[^1]: Luis Caires and Frank Pfenning. 2010. Session Types as Intuitionistic Linear Propositions.
